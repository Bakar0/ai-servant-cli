# Fine-tuning servant instructions

**Status:** implemented (2026-06-16)

> Implementation note: the agent-facing operations are **flags on a single `fine-tune` command**
> (`--list`/`--show`/`--set`/`--scaffold`/`--reset`/`--apply` with a positional `<aspect>`),
> not citty subcommands. This keeps the human entry `servant fine-tune <aspect>` (open a focused
> session) from colliding with subcommand names. Behavior matches the design below.

## Problem

Servant ships its instruction assets as bundled templates and **overwrites** them on
every `init` / `spawn` / `resume`:

- `src/core/claude-setup.ts` → `ensureServantAssets()` copies
  `src/templates/servant_root/**` into `~/.ai_servant/`, rewriting any file whose
  content changed.
- `src/core/extract-prompt.ts` → `buildExtractionPrompt()` is a hard-coded string fed to
  the headless drainer.

A user who hand-edits a delivered asset to customize behavior (e.g. the delegate brief
format, the memory-extraction discipline, the workspace conventions) **loses those edits**
the next time the CLI syncs assets. There is no way to customize *and* keep receiving CLI
updates to the same asset.

## Goal

A `servant fine-tune` command that lets the user customize specific **aspects** of servant's
instructions through per-aspect overlay files that survive CLI asset updates.

Decisions (confirmed):
- **Command name:** `servant fine-tune`.
- **UX:** agent-driven — `servant fine-tune` opens an interactive servant session that interviews
  the user (conversational option-selections) and uses the `servant` CLI to scaffold/write the
  overlay. NOT a `$EDITOR` flow. (See [[feedback_agent_driven_interviews]].)
- **Session host:** a new terminal tab via the cmux/iterm driver (reusing `terminals/`), like
  `servant spawn`.
- **Write path:** the agent pipes the drafted overlay to `servant fine-tune set <aspect>`
  (stdin/`--body-file`), which writes it and recomposes delivered assets in one deterministic step.
- **Override model:** additive — the overlay is appended *after* the base; precedence is by
  natural language ("override the default below: ..."). The base is never forked, so CLI
  updates always flow through; the overlay supersedes by saying so.
- **Scope:** global only — overlays live under `~/.ai_servant/fine-tune/` and apply to every
  workspace. (Per-workspace overlays are a possible future extension, explicitly out of scope.)

## Core model: base + additive overlay

| Layer | Owner | Location | Lifecycle |
|-------|-------|----------|-----------|
| **Base** | CLI | `src/templates/...` + `extract-prompt.ts` | shipped, evolves, overwritten on update |
| **Overlay** | User | `~/.ai_servant/fine-tune/<aspect>.md` | scaffolded once, never clobbered |

Delivered asset = base **with the overlay spliced/appended in**. Composition happens at
sync time inside `ensureServantAssets()` (and at runtime for the headless prompt).

Because overlays are *LLM instructions, not code*, additive composition gives both update-flow
and full override power: a trailing instruction overrides an earlier one just by saying so.
No three-way merge, no conflict markers, no silently-forked sections.

## Aspect model

An **aspect** is one concept the user tunes once; it may feed several delivered assets.

```ts
// src/core/fine-tune.ts
interface TuneAspect {
  id: string;          // "general" | "memory-extraction" | "delegate" | "goal" | "recall"
  title: string;
  blurb: string;       // shown in the picker
  scaffold: string;    // initial overlay body with guidance comments
  consumers: AssetConsumer[]; // delivered files (+ headless prompt) that splice this overlay
}
```

| Aspect | Consumers |
|--------|-----------|
| `general` | root `CLAUDE.md` (reaches every workspace via `@../../CLAUDE.md`) |
| `memory-extraction` | `/servant:extract-memories.md` **and** `buildExtractionPrompt()` — one overlay, both surfaces |
| `delegate` | `/servant:delegate.md` |
| `goal` | `/servant:goal.md` |
| `recall` | `/servant:recall.md` |

The `memory-extraction` row is the reason aspects are logical, not 1:1 with files: the user
tunes "how memories are extracted" once and both the interactive and headless paths pick it up.

## Composition mechanics

**Markdown assets** — sentinel markers in the base template:

```markdown
<!-- servant:tune:start id=delegate -->
<!-- No local fine-tuning yet. Run `servant fine-tune delegate` to customize. -->
<!-- servant:tune:end -->
```

`ensureServantAssets()` reads `fine-tune/delegate.md` and replaces the inner region with the
overlay (or leaves the placeholder when no overlay exists), then writes the composed file.
Idempotency unchanged: compare *composed* bytes vs destination, write only on diff.

**Headless prompt** — `buildExtractionPrompt()` reads `fine-tune/memory-extraction.md` at runtime
and appends a `## Local fine-tuning (user overrides)` section when present.

Overlay scaffold header (teaches precedence):

```markdown
<!-- This file fine-tunes servant's `<aspect>` instructions.
     Everything below is appended AFTER servant's built-in instructions and takes
     precedence where it conflicts. To override a default, say so explicitly. -->
```

## Command UX — agent-driven interview, not `$EDITOR`

`servant fine-tune` does NOT drop the user into an editor. It **opens an interactive servant
session** that interviews the user and uses the CLI to write the overlay — the same shape as a
goal-less `servant spawn` auto-running `/servant:goal` (`spawn.ts` `GOAL_BOOTSTRAP_PROMPT`).

The command is dual-faced, like `servant extract-memories` (hook drainer vs `--reconcile`):

**Human-facing (bare, TTY) — opens the session:**
```
servant fine-tune                 # open a session to fine-tune any aspect
servant fine-tune delegate        # open a session focused on the `delegate` aspect
```
- Opens a **new terminal tab** via the cmux/iterm driver (reusing `terminals/`), `cwd =
  ~/.ai_servant/` (root) so `/servant:fine-tune` resolves and root `CLAUDE.md` auto-loads,
  title `fine-tune`. Global — no workspace required.
- Seed prompt triggers the interview: *"Run /servant:fine-tune to help me customize servant's
  instructions"* (+ the chosen aspect when given).

**Agent-facing (subcommands) — the deterministic write/recompose path the interview calls:**
```
servant fine-tune scaffold <aspect>   # create fine-tune/<aspect>.md w/ guidance header if missing
servant fine-tune set <aspect>        # write overlay body (stdin or --body-file) AND recompose assets
servant fine-tune show <aspect>       # print current effective instructions (base + overlay)
servant fine-tune list                # aspects + "customized?" flag (machine + human readable)
servant fine-tune reset <aspect>      # delete overlay → revert to pure base, recompose
servant fine-tune apply               # recompose all delivered assets (idempotent; called by sync too)
```

**The `/servant:fine-tune` slash command** drives the interview:
1. If no aspect was seeded, present the aspects as a selection.
2. `servant fine-tune show <aspect>` → read back the current effective instructions to the user.
3. Ask, with concrete options, what to change; draft the overlay body.
4. Confirm with the user (nothing persisted until approval — `/servant:goal` discipline).
5. Pipe the approved body to `servant fine-tune set <aspect>`; report which assets changed.

## Implementation footprint

New:
- `src/core/fine-tune.ts` — aspect registry + `composeAsset()`, overlay read/write helpers, `applyAll()`.
- `src/commands/fine-tune.ts` — the command: bare (open session) + `scaffold`/`set`/`show`/`list`/`reset`/`apply` subcommands.
- `src/templates/servant_root/.claude/commands/servant/fine-tune.md` — the interview slash command.
- `fineTuneDir()` (+ `fineTuneAspectPath(id)`) in `src/core/paths.ts`.
- Register `fine-tune` in `src/index.ts`.

Edit:
- `src/core/claude-setup.ts` — `ensureServantAssets()` calls `applyAll()` to compose overlays into markdown assets.
- `src/core/extract-prompt.ts` — append the `memory-extraction` overlay at runtime.
- The 5 template files — add `servant:tune` sentinel markers.
- `src/commands/init.ts` summary line — mention `servant fine-tune`.

Tests (mirror `claude-setup.test.ts`):
- overlay content survives a simulated base update;
- missing overlay → delivered asset equals pure base (placeholder region);
- `set` then `reset` round-trips back to base;
- `set` reads stdin/`--body-file` and recomposes; `apply` is idempotent;
- `memory-extraction` overlay reaches both the slash command and `buildExtractionPrompt()`.

## Edge cases / notes

- **Direct edits to delivered assets** are still clobbered by design — that's the whole reason
  the overlay exists. Optionally detect drift outside the markers and warn the user to move it
  into `servant fine-tune`.
- **`@import` vs splice:** for `general`, native `@import` could work, but markers/splice is
  used uniformly so slash commands and the headless prompt (which cannot import) share one
  mechanism — and to avoid Claude Code's external-import trust prompt for sibling subtrees
  (see the note in `workspace.ts` about not importing the knowledge store).
- **Out of scope:** per-workspace overlays; section-replace override model.
